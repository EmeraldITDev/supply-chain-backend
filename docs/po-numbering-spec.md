# PO Numbering — Backend Specification

Status: **Backend implemented** (Jun 2026). The authoritative PO number (and the
per-supplier-per-day serial) is generated by the backend at PO generation time.
The frontend formatter `src/utils/poNumber.ts` is used for previews/validation
and produces identical output.

## 1. Format

```
PO-DDMMYY-SupplierToken-NNNN
```

| Segment | Rule | Example |
|---------|------|---------|
| `PO` | Literal prefix | `PO` |
| `DDMMYY` | PO creation date, 2-digit day + month + year, no separators | `220626` |
| `SupplierToken` | Supplier name, all non-alphanumerics removed, original casing preserved, ≤30 chars | `MochenzComputers` |
| `NNNN` | 4-digit zero-padded serial, resets per supplier per day | `0001` |

Example: first PO for **Mochenz Computers** on 22 Jun 2026 →
`PO-220626-MochenzComputers-0001`; the second → `PO-220626-MochenzComputers-0002`.

## 2. Supplier token algorithm

Implemented in `PoNumberGenerator::normalizeSupplierToken()` (mirrors frontend
`normalizeSupplierToken`):

1. Remove every non-alphanumeric character (concatenates words).
2. Preserve original casing. Cap at 30 characters.
3. If nothing remains, use `Vendor`.

| Input | Token |
|-------|-------|
| `Mochenz Computers` | `MochenzComputers` |
| `Al-Fatah Trading` | `AlFatahTrading` |
| `3M Nigeria Ltd` | `3MNigeriaLtd` |
| `  &Co Supplies` | `CoSupplies` |
| `!!!` / empty | `Vendor` |

## 3. Serial algorithm

- Scoped to **(supplier_token, creation_date)** via the `po_number_sequences`
  table (`scope_key = "{DDMMYY}|{SupplierToken}"`, `last_serial`).
- Allocated atomically inside a DB transaction with `lockForUpdate`, so
  concurrent generation never yields duplicates.
- First allocation seeds the counter from the highest existing `po_number`
  serial matching the prefix (safe across re-deploys / partial data).
- Final guard: the generated `po_number` is re-checked against `mrfs.po_number`;
  on the rare collision it retries with the next serial.

## 4. Where it applies

| Endpoint | Behaviour |
|----------|-----------|
| `POST /api/mrfs/{id}/generate-po` (finalise) | Assigns + persists `po_number`; returns on `data.mrf.po_number`. |
| `POST /api/mrfs/{id}/generate-po` with `save_as_draft: true` | Does **not** allocate a serial (draft keeps any caller-supplied number only). |
| Regeneration (existing PO, not signed) | Keeps the existing `po_number` (no new serial). |
| Caller-supplied `po_number` | Honoured as-is (frontend preview/legacy path), subject to uniqueness check. |
| `POST /api/trips/{id}/generate-trip-po` | `po_number` now optional; auto-generated from carrier/vendor name when omitted; existing number kept on regeneration. |

Supplier name resolution for MRFs (`MRFWorkflowController::resolveSupplierNameForPo`):
selected price-comparison supplier → MRF `selectedVendor` → `Vendor` fallback.

## 5. Backward compatibility

- Existing PO numbers are **never** rewritten; only POs generated after deploy
  use the new format.
- All display surfaces read `po_number || poNumber` verbatim, so old and new
  numbers render correctly side by side.

## 6. Acceptance criteria

- [x] New PO for Mochenz Computers on 22 Jun 2026 → `PO-220626-MochenzComputers-0001`.
- [x] Second new PO same supplier/day → `...-0002`.
- [x] New PO for Al-Fatah Trading → `PO-220626-AlFatahTrading-0001` (independent serial).
- [x] Serial resets to `0001` per supplier on the next calendar day.
- [x] Concurrent generations never produce duplicate `po_number` (row lock).
- [x] Existing POs retain their original numbers.
- [x] `po_number` returned on generate-po finalise and on `GET /api/mrfs` rows.
